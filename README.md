# CAN Monitor — Техническая документация

Консольное приложение для мониторинга CAN-шины в реальном времени. Подключается
к устройству на базе ESP32 (прошивка GVRET / ESP32RET) через Wi-Fi по TCP.

---

## Сборка

```
cmake -B build -G "MinGW Makefiles"
cmake --build build
```

Или через `Makefile`:
```
make
```

Требования: **C++17**, **Winsock2** (`ws2_32`), Windows 10+.

Запуск с явным IP-адресом устройства:
```
can_monitor.exe 192.168.4.1
```
Без аргументов — сначала UDP-discovery, затем fallback на `192.168.4.1`.

## Главный цикл (`main.cpp`)

В конце каждой итерации — `Sleep(1)`, реальный тик ~1 мс.

### Константы тайминга

| Константа | Значение | Назначение |
|---|---|---|
| `DISPLAY_INTERVAL_MS` | 100 мс | Частота обновления экрана (10 Гц) |
| `KEEPALIVE_MS` | 2000 мс | Интервал keepalive-пакета |
| `RECONNECT_MS` | 3000 мс | Пауза перед повтором подключения |

### Порядок действий за один тик

```
1. Управление соединением
   └─ если нет соединения И прошло ≥ 3 с:
       ├─ UDP-discovery (однократно, блокирует до 2 с)
       ├─ TCP-connect (неблокирующий, таймаут 2 с)
       └─ отправка 4 init-команд

2. Приём данных (неблокирующий)
   └─ recv() до 4096 байт → feedBytes() в парсер

3. Разбор очереди парсера
   └─ пока hasFrame(): store.update(popFrame())

4. Keepalive
   └─ каждые 2 с: отправить F1 09

5. Обновление экрана (каждые 100 мс)
   └─ calculateRates() → display.render()
```

## Соединение (`connection.cpp`)

### UDP-discovery

Приложение слушает **UDP-порт 17222**, ожидая широковещательный пакет от ESP32.

```
Ожидаемый пакет: 4 байта = { 0x1C, 0xEF, 0xAC, 0xED }
IP-адрес отправителя становится целевым адресом.
```

- Блокирует поток на `timeoutMs` = 2000 мс
- Если пакет не пришёл — fallback на `DEFAULT_IP = "192.168.4.1"`
- Выполняется **один раз** за сессию (`discoveryDone = true` после первой попытки)

**Debug:** Discovery не срабатывает, если ESP32 и ПК в разных подсетях
или прошивка ESP32 не отправляет broadcast.

### TCP-соединение

```
socket() → ioctlsocket(FIONBIO=1) → connect() → select() → getsockopt(SO_ERROR)
```

- `connect()` вызывается в неблокирующем режиме
- `select()` ждёт готовности write-сокета, таймаут = 2000 мс
- Сокет **остаётся неблокирующим** после connect — это важно для receive
- При успехе: `connected_ = true`, отправляются 4 init-пакета

**Debug:** Код ошибки WSA отображается в строке статуса.
Типичные значения: `10061` — порт закрыт, `10060` — таймаут, `10065` — нет маршрута.

### Приём данных

```cpp
int n = recv(sock_, buf, maxLen, 0);
// n > 0                → данные получены
// n == 0               → ESP32 закрыл соединение (graceful)
// n < 0, WSAEWOULDBLOCK → данных нет (норма для неблокирующего сокета)
// n < 0, другое        → ошибка → disconnect()
```

Буфер приёма в `main.cpp`: **4096 байт за один вызов recv()**.
В одном recv() может прийти несколько GVRET-фреймов или половина фрейма —
парсер справляется с этим через внутренний конечный автомат.

## Протокол GVRET (`gvret_parser.cpp`)

### Обрамление пакетов

Все пакеты начинаются с двух байт: `0xF1 <команда>`.
Парсер в состоянии `IDLE` ждёт байт `0xF1`, затем читает код команды.

### Команды ESP32 → ПК

| Байт команды | Название | Длина payload |
|---|---|---|
| `0x00` | CAN-фрейм | 9 + DLC + 1 (контрольная сумма) |
| `0x01` | Синхронизация времени | 4 байта |
| `0x02` | Цифровые входы | 2 байта (пропускается) |
| `0x03` | Аналоговые входы | 15 байт (пропускается) |
| `0x06` | Параметры шины | 10 байт |
| `0x07` | Информация об устройстве | 6 байт (используются первые 3) |
| `0x09` | Keepalive | 2 байта |
| `0x0C` | Количество шин | 1 байт |
| `0x0D` | Extended buses | 15 байт (пропускается) |

### Структура CAN-фрейма (команда `0x00`)

После заголовка `0xF1 0x00` парсер накапливает байты в `buf_[32]`:

```
Смещение  Размер  Поле
──────────────────────────────────────────────────────────────
[0..3]      4     Timestamp (little-endian, мс, время работы ESP32)
[4..7]      4     CAN ID (little-endian)
                    бит 31     = флаг extended-фрейма (1=29-бит, 0=11-бит)
                    биты 30:0  = собственно CAN ID
[8]         1     lenBus
                    биты 3:0   = DLC (длина данных, 0..8)
                    биты 7:4   = номер шины (0=CAN0, 1=CAN1)
[9..9+DLC-1] DLC  Байты данных
[9+DLC]     1     Контрольная сумма (принимается, но игнорируется)
```

Полная длина пакета после заголовка `0xF1 0x00`: **10 + DLC** байт
(минимум 10, максимум 18 при DLC=8).

### Конечный автомат парсера

```
0xF1             байт команды
IDLE ──────────► GET_COMMAND ─────────────────────────────────┐
  ▲                                                           │
  │               0x00 → READ_CAN_FRAME                       │
  │               0x01 → READ_TIME_SYNC      (4 байта)        │
  │               0x02 → SKIP_BYTES (2)                       │
  │               0x03 → SKIP_BYTES (15)                      │
  │               0x06 → READ_CANBUS_PARAMS  (10 байт)        │
  │               0x07 → READ_DEV_INFO       (6 байт)         │
  │               0x09 → READ_KEEPALIVE      (2 байта)        │
  │               0x0C → READ_NUMBUSES       (1 байт)         │
  │               0x0D → READ_EXT_BUSES      (15 байт)        │
  │               другое → IDLE                               │
  │                                                           │
  └───────── по завершении любого состояния ──────────────────┘
```

**Переменные состояния:**
- `state_` — текущее состояние (enum)
- `step_` — счётчик байт внутри текущего сообщения (начинается с 0)
- `buf_[32]` — буфер накопления текущего сообщения
- `currentDlc_` — DLC, захваченный на шаге 8 CAN-фрейма

**Ключевой момент в `READ_CAN_FRAME`:**

```cpp
buf_[step_] = b;

if (step_ == 8) {
    currentDlc_ = b & 0x0F;       // DLC из младшего nibble lenBus
    if (currentDlc_ > 8) currentDlc_ = 8;  // защитное ограничение
}

if (step_ >= 9 + currentDlc_) {
    // достигли байта контрольной суммы → фрейм готов
    emitCanFrame();
    state_ = IDLE;
} else {
    step_++;
}
```

`step_` **не инкрементируется** на последнем байте (контрольная сумма).
Это означает, что `buf_[9+DLC]` содержит контрольную сумму, но `emitCanFrame()`
её не читает — используются только `buf_[0..8+DLC]`.

### Сборка CAN-фрейма (`emitCanFrame`)

```cpp
f.timestamp = buf_[0..3]  // little-endian uint32
rawId       = buf_[4..7]  // little-endian uint32
f.extended  = (rawId >> 31) & 1
f.id        = rawId & 0x7FFFFFFF
f.dlc       = buf_[8] & 0x0F
f.bus       = (buf_[8] >> 4) & 0x0F
f.data[i]   = buf_[9 + i]   // i = 0..dlc-1
```

Готовый фрейм помещается в `std::vector<ParsedFrame> frames_`.
Из главного цикла он забирается через `popFrame()` (удаление из начала вектора).

### Команды ПК → ESP32

| Функция | Байты | Назначение |
|---|---|---|
| `buildEnableBinary()` | `E7` | Переключить ESP32 в бинарный режим |
| `buildGetDevInfo()` | `F1 07` | Запросить информацию об устройстве |
| `buildGetBusParams()` | `F1 06` | Запросить параметры шины |
| `buildGetNumBuses()` | `F1 0C` | Запросить количество шин |
| `buildKeepalive()` | `F1 09` | Keepalive-ping |
| `buildTimeSync()` | `F1 01` | Синхронизация времени (не используется) |

**Init-последовательность при подключении:**
```
→ E7          (включить бинарный режим, затем Sleep(50 мс))
→ F1 07       (запрос device info)
→ F1 06       (запрос bus params)
→ F1 0C       (запрос num buses)
```

## Хранилище фреймов (`frame_store.cpp`)

### Структура хранения

```cpp
std::map<uint32_t, FrameEntry> store_;   // ключ — CAN ID
uint64_t total_;                          // суммарное число принятых фреймов
ULONGLONG lastRateCalc_;                  // метка времени последнего пересчёта
```

`std::map` упорядочен по CAN ID — таблица в консоли всегда отсортирована по возрастанию ID.

### Поля `FrameEntry`

```cpp
struct FrameEntry {
    uint32_t  id;            // CAN ID
    bool      extended;      // true=29-бит, false=11-бит
    uint8_t   bus;           // номер шины (0=CAN0, 1=CAN1)
    uint8_t   dlc;           // длина данных
    uint8_t   data[8];       // актуальные данные
    uint8_t   prevData[8];   // данные из предыдущего обновления
    uint64_t  count;         // сколько раз пришёл этот ID
    uint64_t  countSnapshot; // count на момент последнего пересчёта частоты
    double    rate;          // фреймов/с (обновляется раз в 0.5 с)
    ULONGLONG firstSeen;     // GetTickCount64() при первом появлении
    ULONGLONG lastSeen;      // GetTickCount64() при последнем обновлении
    bool      dataChanged;   // true если data изменился в последнем update()
};
```

### Логика `update()`

```
Новый ID:
  → создать FrameEntry
  → dataChanged = true (считается "изменился" при первом появлении)
  → prevData = {0, 0, ...}
  → data = frame.data
  → count = 1

Существующий ID:
  → dataChanged = (memcmp(data, frame.data) != 0)
  → если изменился: prevData ← data
  → data ← frame.data
  → count++, lastSeen = now
```

### Подсчёт частоты (`calculateRates`)

Вызывается каждые 100 мс (из главного цикла), но реально считает **не чаще чем раз в 0.5 с**:

```cpp
double elapsed = (now - lastRateCalc_) / 1000.0;
if (elapsed < 0.5) return;

for (auto& [id, e] : store_) {
    e.rate = (e.count - e.countSnapshot) / elapsed;
    e.countSnapshot = e.count;
}
lastRateCalc_ = now;
```

Это значит:
- Колонка `/s` **обновляется каждые ~0.5 с**, не 10 Гц
- Для нового ID — первые 0.5 с показывается `0`
- Если фрейм пришёл один раз и больше не приходит — rate стремится к 0 после следующего пересчёта

## Отображение (`display.cpp`)

### Инициализация

```cpp
GetStdHandle(STD_OUTPUT_HANDLE)
SetConsoleMode(... | ENABLE_VIRTUAL_TERMINAL_PROCESSING)  // разрешить ANSI на Windows
printf("\033[?25l")      // скрыть курсор
printf("\033[2J\033[H")  // очистить экран, курсор в (1,1)
```

### Последовательность рендера (каждые 100 мс)

```
\033[H              ← курсор в (1,1) без очистки экрана
Строка 1: заголовок "CAN Monitor v1.0"
Строка 2: разделитель (--- ширина терминала)
Строка 3: статус соединения + IP:port
Строка 4: диагностическое сообщение (statusMsg)
Строка 5: модель устройства + скорость CAN
Строка 6: счётчики (всего фреймов, уникальных ID, время работы)
Строка 7: двойной разделитель (=== ширина терминала)
Строка 8: заголовки колонок
Строка 9: разделитель (--- ширина терминала)
Строки 10..N: строки таблицы CAN-фреймов
\033[J              ← очистить от текущей позиции до конца экрана
```

**Почему нет мерцания:** курсор возвращается в начало (`\033[H`) и перезаписывает
строки поверх предыдущих. Каждая строка заканчивается `\033[K` (очистка до конца строки).
`\033[J` (очистка до конца экрана) вызывается **один раз** в самом конце —
только для сдвига пустого пространства при уменьшении числа фреймов.
Прыжков курсора к произвольным строкам нет.

### Расчёт числа отображаемых строк

```cpp
int headerLines = 9;           // строки 1..9 (заголовки)
int maxRows = height - 9;      // остаток — строки таблицы
if (maxRows < 1) maxRows = 1;
```

Высота терминала читается через `GetConsoleScreenBufferInfo()` при каждом render.
Если фреймов больше `maxRows` — лишние отбрасываются (итерация по map обрывается).

### Формат строк таблицы

```
  CAN ID     | DLC | Data                     |  Count  |  /s
  ─────────────────────────────────────────────────────────────
  0x1FF      |  8  | 01 02 03 04 05 06 07 08  | 1000000 |  500
  0x18FF1234 |  4  | AA BB CC DD              |      42 |    1
```

- **CAN ID**: `0x%03X    ` (11-бит, 10 символов) или `0x%08X` (29-бит, 10 символов)
- **DLC**: `| %d   |` — 1 символ, работает для 0..8
- **Data**: hex-строка с ручным padding до 24 видимых символов.
  Изменившиеся байты подсвечиваются жёлтым (`\033[33m`).
  Padding считается по `dlc * 3 - 1` (2 hex + пробел на байт, минус trailing-пробел),
  потому что ANSI-escape коды раздувают длину строки, но не занимают место на экране.
- **Count**: `%7llu` — до 9 999 999
- **/s**: `%4.0f` — целое, до 9999

### ANSI-коды

| Код | Действие |
|---|---|
| `\033[H` | Курсор в позицию (1,1) |
| `\033[K` | Очистить до конца текущей строки |
| `\033[J` | Очистить от курсора до конца экрана |
| `\033[?25l` | Скрыть курсор |
| `\033[?25h` | Показать курсор (при выходе) |
| `\033[0m` | Сброс всех атрибутов |
| `\033[1m` | Жирный |
| `\033[31m` | Красный (Disconnected) |
| `\033[32m` | Зелёный (Connected) |
| `\033[33m` | Жёлтый (изменившиеся байты, statusMsg) |
| `\033[36m` | Голубой (заголовок) |

## Отладка

### Нет соединения

1. Следить за строкой статуса (строка 4) — она показывает каждый шаг подключения.
2. UDP-discovery не срабатывает = нормально, если ESP32 не шлёт broadcast.
   Приложение само перейдёт на `192.168.4.1`.
3. TCP-ошибки:
   - `WSA 10061` — порт 23 закрыт, прошивка не запущена
   - `WSA 10060` — таймаут, проверить Wi-Fi и маршрутизацию
   - `WSA 10065` — нет маршрута, ПК не в сети ESP32-AP

### Подключён, но фреймов нет

1. Строка 5 должна показать `Device: ESP32_RET build XXXX` и `CAN0: XXXXXX bps`
   в течение ~200 мс после подключения. Если нет — init-пакеты потеряны,
   смотреть `INIT FAIL` в строке статуса.
2. ESP32 должен получить `0xE7` для переключения в бинарный режим.
   В текстовом режиме он шлёт ASCII-строки — парсер их игнорирует
   (ждёт `0xF1`, всё остальное отбрасывается в состоянии IDLE).
3. Добавить в `main.cpp` для проверки потока байт:
   ```cpp
   if (n > 0) fprintf(stderr, "RX %d bytes\n", n);
   ```
   Запустить с редиректом: `can_monitor.exe 2>debug.log`

### Отладка парсера

Добавить в начало `processByte()` в `gvret_parser.cpp`:
```cpp
fprintf(stderr, "state=%d step=%d byte=0x%02X\n", state_, step_, b);
```

Типичные признаки проблем:

| Симптом | Вероятная причина |
|---|---|
| Парсер всегда в IDLE | Байт `0xF1` не приходит, ESP32 в текстовом режиме |
| Фрейм застрял в READ_CAN_FRAME | Поток обрывается посередине фрейма, ждёт следующий recv() |
| `emitCanFrame()` не вызывается | DLC = 0? Тогда завершение на шаге 9, проверить buf_[8] |
| Все ID = 0 | rawId всегда 0 — проверить buf_[4..7] в `emitCanFrame()` |

### Частота `/s` всегда 0

- Первые 0.5 с после появления ID — **норма**.
- Если остаётся 0: добавить `fprintf(stderr, "total=%llu\n", total_)` в `update()`.
  Если `total_` не растёт — `hasFrame()` не возвращает true или `update()` не вызывается.

### Проблемы с отображением

- **Колонки съехали**: ширина терминала менее ~55 символов. Расширить окно.
- **Нет цветов / ANSI-коды видны как текст**: запустить из Windows Terminal
  или ConEmu. Старый `cmd.exe` на Windows до версии 1903 не поддерживает
  `ENABLE_VIRTUAL_TERMINAL_PROCESSING`.
- **Курсор виден**: та же проблема с VT-процессингом.

### Keepalive

Отправляется каждые 2 с (`F1 09`). Некоторые прошивки ESP32 разрывают TCP
при отсутствии keepalive. Если соединение стабильно рвётся каждые ~10 с —
проверить что `lastKeepalive` обновляется и `sendData()` возвращает `true`.
